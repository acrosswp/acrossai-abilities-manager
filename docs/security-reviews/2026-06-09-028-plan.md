# SECURITY REVIEW REPORT — PLAN (Feature 028)

## Executive Summary

Feature 028 is a maintenance-only release with no new business logic, no new data models, and no new
user-facing REST endpoints. The primary security surface is the BerlinDB 3.0 / wpb-access-control v1.2.0
upgrade (CHANGE-1), which touches the access control library. The plan correctly mandates
post-upgrade revalidation of SEC-03, SEC-04, and DEC-PERM-CB. One low-severity gap was found:
`DEC-FAIL-OPEN-NOTICE` is omitted from the revalidation checklist in Phase 1.3, even though the project
memory (`DEC-REVALIDATE-SECURITY-POST-UPGRADE`) lists it as a required post-upgrade check.

CHANGE-3 (REST audit) is itself a security activity and its design is sound: the plan mandates checking
all six REST route groups, forbids `__return_true`, and documents the correct two-gate callback pattern.
CHANGE-4 and CHANGE-5 are pure UI changes with no security surface.

**Overall risk: LOW.** One low-severity gap identified; two informational notes.

## Vulnerability Findings

### [LOW] SEC-028-01: `DEC-FAIL-OPEN-NOTICE` omitted from post-upgrade revalidation table

**Finding ID**: SEC-028-01
**CVSS Score**: 2.1 (Low)
**Location**: `specs/028-berlindb-upgrade-rest-audit-ui-fixes/plan.md` — Phase 1.3 revalidation table
**OWASP Category**: A06:2025-Vulnerable and Outdated Components
**CWE**: CWE-306: Missing Authentication for Critical Function

**Description**:
Phase 1.3 of the plan includes a post-upgrade security revalidation table covering SEC-03, SEC-04,
DEC-PERM-CB, and DEC-TABLE-SOFT-SINGLETON. However, the project's durable decision
`DEC-REVALIDATE-SECURITY-POST-UPGRADE` (INDEX.md) lists four required checks including
`DEC-FAIL-OPEN-NOTICE`. That decision mandates that the fail-open library-absence pattern always
pairs with a `manage_options` admin notice — a security-critical control that prevents silent
privilege escalation when `wpb-access-control` is absent.

`wpb-access-control v1.2.0` includes a BerlinDB 3.0 namespace migration. If its class interface
changes break the library-detection mechanism used by the fail-open pattern, permissions would
silently open (abilities accessible without access checks) with no admin notice surfaced.

The omission means the implementation phase has no explicit reminder to verify this control survives
the upgrade.

**Remediation**:
Add a fifth row to the Phase 1.3 revalidation table in `plan.md`:

| Constraint | Verification command | Expected |
|------------|---------------------|----------|
| DEC-FAIL-OPEN-NOTICE (fail-open + notice) | `grep -n "fail.open\|admin_notice\|DEC-FAIL-OPEN" includes/Modules/Abilities/` | Both fail-open path and admin notice hook present and intact |

Also add a corresponding task in `tasks.md` T004 (expand the post-upgrade revalidation step to
include this check).

**Spec-Kit Task**: TASK-SEC-028-01 — Expand T004 post-upgrade revalidation to include DEC-FAIL-OPEN-NOTICE grep

### [INFORMATIONAL] SEC-028-02: phpunit.yml supply chain — `composer install` in matrix

**Finding ID**: SEC-028-02
**CVSS Score**: 0.0 (Informational)
**Location**: `specs/028-berlindb-upgrade-rest-audit-ui-fixes/plan.md` — Phase 2.3 phpunit.yml

**Description**:
The new `phpunit.yml` CI workflow runs `composer install --prefer-dist --no-progress` on all five
PHP matrix runners. This fetches packages from Packagist and GitHub VCS. If the lock file is
compromised (malicious commit), all five runners would execute the tampered dependency.

This is a standard Composer/CI pattern and is mitigated by: (1) the lock file is version-controlled
and any change would be visible in the PR diff; (2) the `actions/checkout` action SHA-pinned per
PATTERN-CI-WORKFLOW-HARDENING; (3) the workflow declares `permissions: contents: read` preventing
secret exfiltration even if a package were malicious.

No action required. Noting for completeness.

### [INFORMATIONAL] SEC-028-03: Post-upgrade revalidation is manual-only

**Finding ID**: SEC-028-03
**CVSS Score**: 0.0 (Informational)
**Location**: `specs/028-berlindb-upgrade-rest-audit-ui-fixes/plan.md` — Phase 1.3

**Description**:
The Phase 1.3 revalidation checks (SEC-03, SEC-04, DEC-PERM-CB, DEC-TABLE-SOFT-SINGLETON) are
documented as manual `grep` commands with no automated CI enforcement. If the upgrade is applied
and the revalidation is skipped (e.g. in a fast-follow patch), the security properties would not
be automatically re-checked.

This is acceptable for a one-time upgrade event. The checks are simple grep patterns that a developer
can execute and document in the PR description. No CI gate is needed for these ad-hoc revalidations.
Noting that this is a deliberate design choice, not an oversight.
